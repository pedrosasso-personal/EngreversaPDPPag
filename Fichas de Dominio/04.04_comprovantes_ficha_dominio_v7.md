## Descrição

O domínio de Comprovantes é responsável por fornecer, armazenar e disponibilizar os dados necessários para a geração de comprovantes de transações financeiras realizadas na plataforma de pagamentos do BV. Este domínio atua na fase pós-transacional, garantindo que todos os pagamentos, transferências, boletos e demais operações possuam dados comprobatórios adequados, em conformidade com requisitos regulatórios e necessidades dos clientes. A responsabilidade pela geração física dos comprovantes (renderização conforme normas corporativas e do BACEN) pertence aos canais de atendimento. O domínio suporta tanto operações de Pessoa Física quanto Pessoa Jurídica, abrangendo todos os tipos de transações disponíveis na plataforma (PIX, TED, TEF, boletos, tributos, débito automático e pagamentos em lote), incluindo operações massivas originadas por parceiros BaaS.

## Síntese executiva

O domínio de Comprovantes desempenha papel estratégico na experiência do cliente e na conformidade regulatória da plataforma de pagamentos, atuando como provedor centralizado de dados comprobatórios. Sua importância se manifesta em quatro dimensões principais: (1) **Conformidade Regulatória** - atendimento às exigências do Banco Central quanto ao armazenamento e disponibilização de registros de transações, com prazos de retenção variando de 2 a 10 anos conforme o tipo de operação, incluindo dados específicos de segurança e incidentes conforme Manual de Segurança do PIX; (2) **Experiência do Cliente** - fornecimento imediato de dados para comprovação de pagamentos realizados através de qualquer canal, essencial para a confiança e satisfação dos usuários; (3) **Rastreabilidade Operacional** - suporte a processos de conciliação, contestação e investigação de transações através do fornecimento de dados estruturados e completos; (4) **Habilitação BaaS** - capacidade de fornecer dados de comprovantes para transações originadas por parceiros, com identificação clara da origem e APIs específicas para consulta.

O domínio integra-se diretamente com o domínio de Transação e Processamento (02.03) para receber dados das operações executadas, com os Canais (01.01 e 01.02) para disponibilizar dados de comprovantes, com o domínio de Logs e Auditoria (05.02) que é responsável pela auditoria e rastreabilidade das operações, e com sistemas de armazenamento corporativos para garantir a retenção de longo prazo. A arquitetura deve suportar disponibilização de dados em formato estruturado (JSON, XML) para que diferentes canais possam gerar comprovantes em seus formatos específicos (PDF, texto, imagem, HTML), garantindo multicanalidade e consistência de informações independente do canal utilizado.

Para operações corporativas e BaaS, o domínio oferece capacidades específicas de processamento em lote, com APIs otimizadas para consulta massiva, exportação estruturada e mecanismos de filas para solicitações de alta volumetria, suportando cenários de 10.000+ transações por requisição.

Este domínio é fundamental para o posicionamento do BV como banco relacional e provedor BaaS, pois dados de comprovantes acessíveis e bem estruturados aumentam a transparência e fortalecem a confiança dos clientes e parceiros, habilitando maior engajamento e uso recorrente da plataforma de pagamentos através de qualquer canal (próprio ou parceiro).

## Elementos imprescindíveis

- Dados obrigatórios de comprovantes conforme Resolução BCB (CPF/CNPJ, ISPB, agência, tipo de conta, número da conta)
- Informações específicas por tipo de transação (end-to-end ID para PIX, número de autenticação para TED, código de barras para boletos)
- Timestamp das transações em fuso horário UTC
- Identificação única do comprovante (ID do comprovante)
- Status da transação (aprovada, rejeitada, em processamento)
- Valores envolvidos (valor principal, tarifas, tributos)
- Dados do pagador e recebedor
- Canal de origem da transação
- Identificação do parceiro BaaS quando aplicável
- Número do protocolo/autorização
- Dados de assinatura digital para validação de autenticidade
- Capacidade de fornecer dados em formato estruturado (JSON, XML) para consumo pelos canais
- APIs de consulta individual e em lote
- APIs otimizadas para exportação massiva (10.000+ registros)
- Mecanismo de paginação eficiente para grandes volumes
- Capacidade de processamento assíncrono via filas para solicitações massivas
- Armazenamento seguro com criptografia
- Controle de acesso autenticado aos dados de comprovantes
- Mecanismo de consulta de dados de comprovantes por múltiplos critérios
- Capacidade de fornecer dados para reemissão (segunda via)
- Versionamento de dados de comprovantes em caso de alterações
- Marcação de fraude quando aplicável
- Dados de conciliação contábil
- Informações de Split Payment quando aplicável
- Certificados digitais para validação histórica (incluindo desativados)
- Integração com sistemas legados para recuperação de histórico
- Suporte a diferentes perfis de cliente (PF, PJ Low Corp, Mid Corp, Large Corp)
- Capacidade de fornecer dados em lote para operações corporativas (formato CNAB 240 quando aplicável)
- Conformidade com prazos de retenção regulatórios (2 anos para consultas, 10 anos para escritas no DICT)
- Estrutura de dados padronizada para diferentes tipos de transação (ISO 20022, CNAB 240)
- Dados de cabeçalhos HTTP das requisições (quando aplicável)
- Informações de conteúdo XML de requisições e respostas (quando aplicável)
- Dados de requisições de ataques de varredura ou incidentes de cibersegurança (retenção de 10 anos)
- Informações de consultas de chave PIX ao DICT ou base interna
- Dados de requisições de escrita ao DICT (criação, portabilidade, notificação de infração, marcação de fraude)
- SLA definido para geração de dados em lote
- Capacidade de identificação e segregação de transações por origem (canal próprio vs parceiro BaaS)

## Integração entre domínios

- **Transação e Processamento (02.03)**: Recebe dados de transações executadas para armazenamento e disponibilização; fornece confirmação de armazenamento de dados de comprovante; recebe identificação de origem (canal próprio ou parceiro BaaS)
- **Canais Próprios (01.01) e Canais Parceiros (01.02)**: Fornece dados estruturados de comprovantes para que os canais gerem a representação visual conforme suas necessidades; recebe solicitações de consulta de dados individual e em lote; fornece APIs específicas para parceiros BaaS consultarem comprovantes de suas transações
- **Notificações e Relatórios (04.05)**: Fornece dados de comprovantes para que sejam enviados aos clientes via diferentes canais; recebe solicitações de dados para compartilhamento
- **Validações de Pagamento (02.02)**: Recebe informações de validações realizadas para armazenamento junto aos dados do comprovante
- **Instrumentos de Pagamento (02.04)**: Integra dados de boletos emitidos e QR Codes para disponibilização em comprovantes de emissão
- **Processamento de Recebimento (02.05)**: Recebe confirmação de pagamentos recebidos para armazenamento de dados de comprovantes de recebimento
- **Tarifas (04.08) e Tributos (04.09)**: Obtém valores de tarifas e tributos aplicados para inclusão nos dados do comprovante
- **Autorizações (04.07)**: Recebe número de autorização e dados de aprovação para inclusão nos dados do comprovante
- **Logs e Auditoria (05.02)**: Fornece eventos de armazenamento e consulta de dados de comprovantes para registro em trilha de auditoria; o domínio de Logs e Auditoria é responsável pela rastreabilidade e auditoria das operações
- **Dados e Monitoramento (05.01)**: Fornece métricas de armazenamento, consulta e disponibilização de dados de comprovantes, incluindo métricas específicas de operações em lote e performance de APIs
- **Iniciação de Pagamentos (06.01)**: Fornece dados de comprovantes de pagamentos iniciados via Open Finance
- **Compartilhamento de Dados (06.02)**: Disponibiliza dados de transações para compartilhamento via Open Finance
- **Sistemas Legados (integração externa)**: Recupera histórico de dados de comprovantes de transações anteriores à plataforma unificada
- **Agendamento e Recorrência (02.01)**: Recebe dados de transações agendadas e recorrentes para armazenamento de comprovantes

## Macro Requisitos

- **MR-04.04-001** — Armazenamento Centralizado de Dados de Comprovantes: O sistema deve armazenar de forma centralizada todos os dados necessários para comprovação de transações financeiras executadas na plataforma (PIX, TED, TEF, boletos, tributos, débito automático, pagamentos em lote), incluindo todas as informações obrigatórias regulatórias e dados específicos de cada tipo de transação, em tempo real após a conclusão da operação, com identificação clara da origem (canal próprio ou parceiro BaaS).

- **MR-04.04-002** — Conformidade com Requisitos Regulatórios de Retenção: O sistema deve armazenar e manter disponíveis os dados de comprovantes em conformidade com os requisitos do Banco Central, mantendo registros por no mínimo 2 anos para transações de consulta e 10 anos para operações de escrita no DICT, com capacidade de fornecimento descriptografado no formato requerido pelo BC quando solicitado.

- **MR-04.04-003** — Disponibilização de Dados para Canais: O sistema deve fornecer dados estruturados de comprovantes através de APIs para que os diferentes canais (próprios e parceiros) possam gerar a representação visual dos comprovantes conforme suas necessidades específicas e padrões de identidade visual, garantindo consistência de informações independente do canal.

- **MR-04.04-004** — Consulta e Recuperação de Dados: O sistema deve permitir consulta e recuperação de dados de comprovantes de transações através de APIs, com performance adequada e suporte a diferentes critérios de busca, incluindo consultas históricas dentro dos prazos de retenção regulatórios.

- **MR-04.04-005** — Segurança e Autenticidade dos Dados: O sistema deve garantir a segurança dos dados de comprovantes através de armazenamento criptografado, controle de acesso autenticado, assinatura digital para validação de autenticidade, e manutenção de certificados digitais (incluindo desativados) para consulta histórica e validação de assinaturas.

- **MR-04.04-006** — Fornecimento de Dados para Reemissão: O sistema deve fornecer dados completos para que os canais possam gerar segunda via de comprovantes, mantendo a capacidade de recuperação de dados históricos conforme necessário e garantindo consistência com o comprovante original.

- **MR-04.04-007** — Suporte a Operações Corporativas: O sistema deve atender necessidades específicas de clientes PJ, incluindo fornecimento de dados em lote para pagamentos múltiplos, estrutura de dados adequada para conciliação automatizada (formato CNAB 240 quando aplicável), controle de acesso por alçadas, e APIs para integração com sistemas corporativos de gestão financeira.

- **MR-04.04-008** — Integração com Histórico Legado: O sistema deve integrar-se com sistemas legados para recuperação de dados de comprovantes históricos anteriores à plataforma unificada, garantindo continuidade de acesso aos clientes e conformidade com prazos regulatórios de retenção.

- **MR-04.04-009** — Escalabilidade e Performance: O sistema deve suportar alta volumetria de armazenamento e consulta de dados de comprovantes, considerando que cada transação da plataforma gera pelo menos um conjunto de dados de comprovante, com capacidade de processamento em tempo real e consultas performáticas mesmo em períodos de pico de operações (até 15.000 transações por segundo).

- **MR-04.04-010** — Padronização de Estrutura de Dados: O sistema deve fornecer dados de comprovantes em formatos estruturados padronizados (JSON, XML) com esquemas bem definidos para cada tipo de transação, incluindo dados obrigatórios conforme padrões ISO 20022 (para PIX) e CNAB 240 (para operações em lote), facilitando o consumo pelos diferentes canais e garantindo consistência na apresentação das informações aos clientes.

- **MR-04.04-011** — Armazenamento de Dados de Segurança e Incidentes: O sistema deve armazenar dados específicos de segurança conforme Manual de Segurança do PIX, incluindo cabeçalhos HTTP das requisições e respostas, dados de assinatura digital, requisições que tenham sido geradas a partir de ataques de varredura ou incidentes de cibersegurança (por 10 anos), e todas as requisições referentes a consultas de chave PIX e transações PIX (por 2 anos).

- **MR-04.04-012** — Processamento Massivo de Dados de Comprovantes: O sistema deve suportar geração e consulta massiva de dados de comprovantes (10.000+ transações por requisição), com APIs otimizadas para operações em lote, formato estruturado para exportação massiva, paginação eficiente para grandes volumes, e SLA definido para processamento de solicitações massivas através de mecanismo de filas assíncronas.

- **MR-04.04-013** — Suporte a Operações BaaS: O sistema deve fornecer capacidades específicas para parceiros BaaS, incluindo identificação clara da origem das transações (parceiro específico), APIs dedicadas para parceiros consultarem comprovantes de suas transações com controle de acesso segregado, e dados estruturados que permitam aos parceiros aplicarem sua própria identidade visual nos comprovantes gerados.
